# Spec: Generative Puzzles

> **TL;DR (read this first):** Add AI-generated word puzzles (scramble, cryptogram, fill-in-the-blank) as a new `messages.type` value (`'puzzle'`), inserted by the narration AI when a story beat calls for one. Per-story budget driven by length (25→2 / 50→2 / 100→4 / 200→6 targets, stored in `app_config` for admin tuning). Three-tier hints, optional skip with soft narrative continuation — puzzles are *moments*, never gates. Template-driven generation with programmatic solvability validation; the project's `haiku` model alias generates puzzles, narration stays on the `sonnet` alias (per current `aiModel.ts` resolution). First mode under the "Story Mode beyond reading" pillar (siblings: AR/location, voice chat, in-character texts — all parked for later). Drafted 2026-05-18; revised same day after spec-document-reviewer pass.

---

## Context & motivation

The `project_future_brainstorms` memory has carried three big-direction ideas for months: audio drama, AI-generated puzzles, walk-to-earn-style mechanics. Today's brainstorm (2026-05-18) expanded that list to a **fourth** — in-character texts that reach the reader in their daily life — and surfaced the connective tissue: all four are ways the story stops being words-on-a-screen and becomes something the reader *does*. Working name for the pillar: **Story Mode beyond reading**.

After weighing compositional (design all four for shared architecture from day one) vs. piecemeal (ship one, learn, then pick the next), the user chose piecemeal — the live product (`mystorymode.com`) is still finding its shape, vision will keep evolving, and three of the four modes need infrastructure that barely overlaps anyway. The umbrella is kept as *direction-setting* (a north star for what kind of feature each new idea is), not as architecture.

**Puzzles go first** because they are the smallest infra change of the four (ship inside the current chat UI), play to the AI's biggest strength (language), and validate the core question for the whole pillar: do readers want the story to *ask something of them* beyond a choice?

### Constraints carried from memory + codebase audit

- **Solvability risk is the central engineering problem.** Raw LLM puzzle generation is often unsolvable or trivial. Mitigation: template-driven generation + programmatic verification before showing the puzzle to a reader.
- **Don't lock in compositional architecture for the other three modes.** Do build the *one* tiny seam that doesn't preclude them — puzzles ship as a new `messages.type` value, not as a special-cased field on existing rows.
- **Stories are short.** A "page" is one AI message + one player choice cycle. The four lengths are 25 / 50 / 100 / 200 pages.
- **Cost discipline matters.** Daily spend cap (`$10/day` per `dailySpend` table) is real; per-puzzle cost must be ~$0.01 or less.
- **Narration voice rules in `docs/ai-voice.md` apply to narration only.** Puzzle content is generated by a separate call with its own prompt and is not subject to the narration voice constraints (no em-dash strip, no banned-vocabulary list).
- **AI access path is OpenRouter via the OpenAI SDK** (`server/aiService.ts` line 44; `baseURL: "https://openrouter.ai/api/v1"`), not the Anthropic SDK directly. Model selection flows through `MODEL_ALIASES` in `server/aiModel.ts`: `haiku` → `anthropic/claude-3.5-haiku`, `sonnet` → `anthropic/claude-sonnet-4`. **Use aliases throughout this spec** — version IDs evolve.
- **All IDs are UUID `varchar`** (`gen_random_uuid()::text`), never serial integers. Every new table follows this convention.
- **RLS is enabled on every table with no policies** (per `reference_supabase_rls` memory + `migrations/0003_issue_reports.sql` template). The Express server uses the `postgres` role and bypasses RLS; anon/authenticated roles are denied.

### Non-goals (v1)

- **Visual puzzles** — hidden objects, jigsaw, spatial reasoning. Need image generation + interactive overlay. Punted to v2/v3.
- **Memory/cognition puzzles** — matching, Simon, keypad sequences. Don't really need an LLM; lower priority for the generative thesis.
- **Cross-puzzle chaining** — answer of puzzle 1 unlocks puzzle 3. Interesting but adds chain-failure risk. v2.
- **Reader-triggered "Challenge me" button.** Considered; rejected for v1 because it loses the magic of narration *saying* "a cipher is carved into the wall."
- **Authored-beat templates in the wizard** ("include puzzles?" toggle). Considered; rejected for v1 because it's rigid and complicates the wizard.
- **AR, voice chat, in-character texts** — sibling pillar modes, parked for dedicated future brainstorms.

## Architecture overview

Two-stage AI flow built around a new `messages.type` value.

```
1. Reader makes a choice → /api/story/respond (or current narration endpoint) fires
                                    ↓
2. Narration AI call (sonnet alias)
   Returns: { narration: "You find a chest with scrambled letters carved into it...",
              puzzle_request?: { type, theme, difficulty } }
                                    ↓
3. Server inspects response. If puzzle_request is present AND we're under the cap:
                                    ↓
4. Puzzle generation call (haiku alias) with type-specific template prompt
   Returns: { answer, payload (client-visible), hints: [hint1, hint2, hint3] }
                                    ↓
5. Server VALIDATES (type-specific, deterministic):
   - scramble: sorted(payload.letters) === sorted(answer)
   - cryptogram: applying payload.mapping to payload.ciphertext === answer
   - fill-in-the-blank: answer fits the blank slot, length plausible
   On failure → re-roll once → fall back to a pre-written puzzle from a hand-curated pool
                                    ↓
6. Persist:
   - INSERT into puzzles (answer + hints server-side only)
   - INSERT into messages with type='puzzle', sender='system', content='',
     puzzle_id pointing at the new puzzle row
                                    ↓
7. Client renders narration bubble + PuzzleCard (no answer, no hints initially)
                                    ↓
8. Reader interacts:
   - "Hint" tap → client reveals next pre-loaded hint tier
   - Submit answer → POST /api/puzzle/attempt → server validates → returns correct/incorrect
   - "Skip Puzzle" tap → POST /api/puzzle/attempt with skip=true
                                    ↓
9. Resolution flows into the NEXT narration call's context — see §Approach 6 for
   the precise threading mechanism (no new column required; the resolver pulls
   recently-resolved puzzles for the current story alongside the existing
   recent-messages window).
```

The narration AI is *not* aware of the puzzle's answer or solution logic — it only sees the resolution signal on the next turn. This isolates the puzzle subsystem so failures (gen errors, validation misses) can't poison narration.

## Approach 1 — Add a `type` column to `messages`

### Goal

Honor the message-type seam decided in brainstorming so that future "Story Mode beyond reading" modes (texts, AR, voice) can each register as another type without restructuring the chat.

### Why a new column (not overloading `sender`)

The reviewer correctly flagged that **`messages` today has no `type` column and no enum** — it has `sender: text("sender").notNull()` (`shared/schema.ts:65`) with values `'dm' | 'npc' | 'player'`. Three options were considered:

| Option | Pros | Cons | Verdict |
|---|---|---|---|
| **A. Add `type` column.** Values `'chat' \| 'puzzle'` (extensible) | Semantically clean — `type` answers "what kind of message" and `sender` answers "who said it." Future modes (AR cue, in-character text, voice cue) add new `type` values without semantic strain. | One new column; existing inserts must explicitly set `type='chat'` (default handles it). | **Chosen.** |
| **B. Overload `sender` with `'puzzle'`.** | Smallest schema change (no new column). | Semantically wrong — a puzzle isn't a "who." Breaks code that filters `sender IN ('dm','npc')` to find AI-authored content. Doesn't extend naturally to AR / voice / text modes. | Rejected. |
| **C. Separate `puzzle_events` table** merged into the chat by timestamp. | Maximum separation; puzzle code never touches `messages`. | Two streams to merge in client + server; every future mode needs its own table; chat-ordering bugs become possible. | Rejected. |

### Schema change

**Drizzle (`shared/schema.ts`):**

```ts
export const messages = pgTable("messages", {
  id: varchar("id").primaryKey().default(sql`gen_random_uuid()`),
  sessionId: varchar("session_id").notNull(),
  storyId: varchar("story_id"),
  content: text("content").notNull(),
  sender: text("sender").notNull(),
  senderName: text("sender_name"),
  timestamp: text("timestamp").notNull(),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
  // NEW: discriminator for non-chat message kinds. Existing rows are 'chat'.
  type: text("type").default('chat').notNull(),
  // NEW: FK populated only when type='puzzle'. Set to NULL on the puzzle row's
  // deletion so the message row survives audit even if the puzzle is wiped.
  puzzleId: varchar("puzzle_id"),
});
```

**Migration (`migrations/0004_puzzles.sql`):**

```sql
ALTER TABLE messages ADD COLUMN IF NOT EXISTS type TEXT NOT NULL DEFAULT 'chat';
ALTER TABLE messages ADD COLUMN IF NOT EXISTS puzzle_id VARCHAR;
-- FK added separately after puzzles table is created (see §Data model).
```

For `type='puzzle'` rows: `content=''`, `sender='system'` (sentinel for non-authored messages — does NOT collide with existing `'dm'|'npc'|'player'` values), `puzzle_id` populated.

### Client render switch (`ChatInterface.tsx`)

Today `getSenderBadge(sender, senderName)` (lines 456–465) is the dispatch. Change: at the render-message layer (above the badge), branch on `type` first:

```tsx
switch (msg.type) {
  case 'puzzle':
    return <PuzzleCard puzzleId={msg.puzzleId!} />;
  case 'chat':
  default:
    // existing GuideBubble / PlayerBubble dispatch on sender
    return msg.sender === 'player'
      ? <PlayerBubble msg={msg} />
      : <GuideBubble msg={msg} />;
}
```

### Implementation surface

- `shared/schema.ts` — add `type` + `puzzleId` columns; regenerate `insertMessageSchema` (createInsertSchema picks up the defaults automatically — verify the omit/partial derivatives still typecheck).
- `client/src/components/ChatInterface.tsx` — add the `type` branch above the existing sender dispatch.
- All server inserts into `messages` continue to omit `type` (gets the `'chat'` default); only the new puzzle-insert path sets `type='puzzle'`.

### Trade-offs

- **Pro:** Clean seam; minimal blast radius on existing code (every existing insert keeps working without modification thanks to the default).
- **Pro:** Future modes (text-from-character, AR-trigger, voice-cue) slot in as additional `type` values without further schema churn.
- **Con:** Adds a nullable `puzzle_id` column only populated for one type. Acceptable — the alternative (polymorphic JSON in `content`) is messier and breaks the existing render contract.

## Approach 2 — Two-stage AI flow (narration + puzzle gen)

### Goal

Let the narration AI request a puzzle organically without burdening it with generating valid puzzle structure. Isolate the most failure-prone work (generation + validation) so it can't break narration.

### Design

**Narration call (existing, with prompt addition).** The narration call already uses `resolveModel(modelOverride)` (four call sites in `server/aiService.ts`: 515, 965, 1083, 1198). Add the following to the user-context block (not the system prompt — these are per-story dynamic values):

- `puzzle_count_so_far` (int) — total puzzles inserted in this story
- `puzzle_target` (int) — target for this story length
- `puzzle_cap` (int) — hard cap; if reached, the prompt forbids emitting `puzzle_request`
- `current_progress` (float 0.0–1.0) — for difficulty signaling
- `recent_puzzle_resolutions` (array of strings) — see §Approach 6 below

Narration JSON schema gains an optional field:
```json
{
  "narration": "...",
  "choices": [...],
  "puzzle_request": {           // optional
    "type": "scramble" | "cryptogram" | "fill-in-the-blank",
    "theme": "pirate treasure",
    "difficulty": "easy" | "medium" | "hard"
  }
}
```

System prompt addition (rough shape, conditional on `puzzles_enabled` flag — see §Rollout):
> When a narrative beat naturally calls for the reader to *figure something out* — a cipher carved on a wall, a torn document with a missing word, an inscription on a found object — you MAY include a `puzzle_request` object alongside your narration. Choose the type that fits the beat: `cryptogram` for ciphers/coded messages, `scramble` for engravings/inscriptions, `fill-in-the-blank` for torn/burned documents. Set `theme` from current story context. Set `difficulty` based on `current_progress`. **You may include a puzzle ONLY if** `puzzle_count_so_far < puzzle_cap`. Aim to land near `puzzle_target` across the story. Do not include `puzzle_request` if it would feel forced.

**Puzzle-generation call (new, separate).** Uses the `haiku` alias regardless of admin overrides — puzzle generation is structural, not narrative, and the admin model toggle exists to A/B narration quality. Returns strict JSON:
```json
{
  "answer": "TREASURE",
  "payload": { "letters": "AETRESUR" },    // shape varies by type
  "hints": ["something a sailor seeks", "starts with T", "T_E_S_R_"]
}
```

### Model resolution — extend `resolveModel`, don't bypass it

The reviewer correctly flagged that `resolveModel(headerValue: string | undefined): string` is the current signature (`server/aiModel.ts:117`) and that all four narration call sites pass `modelOverride` positionally. **Do not break those call sites.** Extend backward-compatibly:

```ts
// server/aiModel.ts — extended signature
export function resolveModel(
  headerValue: string | undefined,
  opts?: { purpose?: 'narration' | 'puzzle-generation' }
): string {
  // For puzzle generation, ALWAYS return the haiku alias resolved to its full
  // ID — bypassing dev header, admin override, and env override entirely.
  // Rationale: puzzle gen is a structural task; the admin model toggle is for
  // A/B-testing narration voice. Keeping puzzles deterministic on Haiku means
  // (a) cost predictability and (b) the admin can A/B narration without
  // unintentionally swapping out the puzzle generator.
  if (opts?.purpose === 'puzzle-generation') {
    return aliasToId('haiku');
  }
  // Existing behavior: dev header → admin override → env → DEFAULT_MODEL.
  // ... (unchanged from current implementation)
}
```

All four existing narration call sites continue to work without modification (opts is optional). New puzzle code in `server/puzzleService.ts`:

```ts
const model = resolveModel(undefined, { purpose: 'puzzle-generation' });
```

### Trade-offs

- **Pro:** Narration prompt stays focused on narrative; puzzle prompt stays focused on structure.
- **Pro:** Haiku is materially cheaper than Sonnet (current OpenRouter pricing: Haiku ~$0.80/1M input vs Sonnet ~$3/1M; output ratio similar) and faster (lower TTFB) — material for both cost and latency.
- **Pro:** If puzzle gen fails twice and fallback fails, we drop the puzzle silently; narration is already rendered.
- **Pro:** Single seam preserved — every model decision still flows through `resolveModel`.
- **Con:** Adds an extra round trip per puzzle (sub-second on Haiku). Acceptable; the visual flow already accommodates a small pause between narration and puzzle card.
- **Con:** The narration prompt grows. Mitigate by gating the new section behind a `puzzles_enabled` `app_config` flag so we can measure narration regression before broad rollout (see §Rollout).

### Implementation surface

- `server/aiModel.ts` — extend `resolveModel` signature with optional `opts.purpose`; add the `purpose === 'puzzle-generation'` early-return.
- `server/aiService.ts` — extend narration system prompt with the puzzle_request section (gated on `puzzles_enabled`); add `puzzle_request` to the Zod parse schema for narration responses; thread budget context into the user-prompt block.
- `server/puzzleService.ts` — **NEW.** Generation prompts per type, `generatePuzzle(req)`, `validatePuzzle(p)`, fallback pool loader.

## Approach 3 — Solvability validation

### Goal

Guarantee that no puzzle reaches a reader unless it has been deterministically verified solvable.

### Design

Each puzzle type has a validator function. The validator runs server-side immediately after generation and re-rolls on failure.

| Type | Validator |
|---|---|
| **Scramble** | `sorted(payload.letters.toLowerCase()) === sorted(answer.toLowerCase())` AND `answer.length` between 4 and 9 AND `answer` is in a curated wordlist (~10k common English nouns/objects bundled with the server). |
| **Cryptogram** | Apply `payload.mapping` to `payload.ciphertext`; assert equals `answer`. Assert `answer.length` 8–40 chars. Assert mapping is a bijection over A–Z (no two ciphertext letters map to the same plaintext letter). Assert at least one letter in the mapping is pre-revealed in `payload.revealed` (a hint letter, so the puzzle isn't pure brute force). |
| **Fill-in-the-blank** | `payload.sentence.includes('___')` exactly once; `answer.length` ≥ 3; `answer` plausibly fits the blank length range hinted in `payload`. |

**Failure handling at generation time:**
1. Validation fails → re-roll once with a slightly stricter prompt (`"the previous attempt failed validation: <reason>"`)
2. Re-roll fails → pull from a hand-curated fallback pool (≥20 pre-written puzzles per type, themed generic) keyed by type + difficulty
3. Fallback pool empty (shouldn't happen) → log warning to Sentry, skip the puzzle entirely (narration already rendered, no user impact)

**Themed-story limitation on wordlist:** the wordlist constraint on scramble validation rejects uncommon themed words (e.g., "ASTROLABE" in a sci-fi story, made-up fantasy proper nouns). Mitigation: on first failure, re-roll with the prompt instructing the AI to pick a more common synonym from the theme. If the second attempt also misses the wordlist, fall back. We accept the trade-off — better to use "TELESCOPE" instead of "ASTROLABE" than to ship an unverifiable scramble.

### Implementation surface

- `server/puzzleService.ts` — validators, fallback pool loader.
- `server/puzzles/wordlist.txt` — **NEW.** Bundled wordlist for scramble validation (~10k entries). Source: a public-domain English-noun frequency list (e.g., the Project Gutenberg derived lists or `wordfreq`'s top-10k common-noun export). Loaded once at module init, kept in a `Set<string>` for O(1) lookup. License: PD; document the source in a comment at the top of the file.
- `server/puzzles/fallback/*.json` — **NEW.** Fallback puzzle pool, one file per type, 20+ entries each, each entry pre-validated.

### Trade-offs

- **Pro:** Deterministic correctness. Readers will never see an unsolvable puzzle.
- **Pro:** Fallback pool guarantees liveness even if AI generation has a bad day.
- **Con:** Curating 60+ fallback puzzles (3 types × 20) is one-time work but real. Estimate: 2–3 hours.
- **Con:** The wordlist constraint constrains themed creativity. Acceptable per the mitigation above; revisit if data shows the fallback rate is dominated by wordlist misses.

## Approach 4 — Per-story budget + difficulty arc

### Goal

Pace puzzles so they feel like special moments, not constant interruptions. Scale appropriately across the four story lengths.

### Design

Budgets stored in `app_config` (existing table — see `shared/schema.ts:192`) under key `puzzle_budgets`, JSON value:

```json
{
  "25":  { "target": 2, "cap": 2 },
  "50":  { "target": 2, "cap": 3 },
  "100": { "target": 4, "cap": 5 },
  "200": { "target": 6, "cap": 8 }
}
```

### Loading + caching pattern (matches `aiModel.ts`, not a new TTL pattern)

The existing admin-model-toggle pattern in `aiModel.ts` is: **module-level variable + one-shot DB load at boot + synchronous in-memory update from the admin POST handler**. There is no 30-second TTL anywhere. Match it precisely:

```ts
// server/puzzleConfig.ts (NEW)
let _budgets: Record<string, { target: number; cap: number }> | null = null;

export function getBudgets() {
  return _budgets;  // null until loadBudgets() runs at boot
}

export function setBudgets(next: Record<string, { target: number; cap: number }>) {
  _budgets = next;  // called synchronously by the admin POST after DB write
}

export async function loadBudgets(storage: IStorage): Promise<void> {
  const row = await storage.getConfig('puzzle_budgets');
  if (row?.value) {
    try { _budgets = JSON.parse(row.value); }
    catch { _budgets = DEFAULT_BUDGETS; }
  } else {
    _budgets = DEFAULT_BUDGETS;
  }
}

const DEFAULT_BUDGETS = { "25": { target: 2, cap: 2 }, ... };  // fallback if DB empty
```

Wired into `server/index.ts` boot the same way `loadAdminModelOverride` is wired today.

### Difficulty arc

Derived, not configured:
- For stories with `target >= 3` puzzles: easy until `current_progress < 0.4`, medium 0.4–0.7, hard ≥ 0.7
- For stories with `target < 3` puzzles: medium throughout

`current_progress` = `currentPage / totalPages`. Already on `gameState`.

### Implementation surface

- `shared/schema.ts` — no change (uses existing `app_config`).
- `server/puzzleConfig.ts` — **NEW.** Module-level cache + load/set helpers, matching the `aiModel.ts` admin-override pattern.
- `server/index.ts` — call `loadBudgets(storage)` at boot, sibling to `loadAdminModelOverride`.
- `server/aiService.ts` — read budget from `getBudgets()` synchronously when building narration context; count existing puzzles via `storage.countPuzzlesForStory(storyId)`; inject the four budget fields plus `current_progress` into narration prompt every turn.
- `client/src/components/AdminDashboard.tsx` — *deferred to a follow-up.* For v1, admins edit `puzzle_budgets` via `app_config` SQL update; an admin dashboard card can come later.

### Trade-offs

- **Pro:** Tunable post-launch from real reader data without redeploy.
- **Pro:** Hard cap protects against AI ignoring soft target.
- **Pro:** Matches the existing well-understood admin-config pattern — no new caching primitive.
- **Con:** Module-level singleton means horizontal scaling needs each instance to load on boot. Already true for `aiModel.ts` admin override; admin updates currently only propagate to other instances on their next boot. This is a known project posture, not a new issue.

## Approach 5 — Hints (bounded, three tiers)

### Goal

Reduce frustration without breaking the puzzle. Three tiers, decreasing puzzle-ness.

### Design

Hints are generated in the **same haiku call** as the puzzle itself — no extra cost, no per-tap latency. All three ship to the client with the puzzle payload.

| Type | Hint 1 | Hint 2 | Hint 3 |
|---|---|---|---|
| **Scramble** | Category ("something a sailor needs") | First letter ("starts with T") | Length + every other letter ("T\_E\_S\_R\_") |
| **Cryptogram** | Theme of the message | One more letter mapping revealed | Reveal a common word ("contains 'KEY'") |
| **Fill-in-the-blank** | Category of missing word | First letter | First and last letter |

Generation prompt for each type explicitly requests these three hint shapes; validators check that hints don't trivially leak the answer (e.g., hint 1 must not contain the answer word itself, hint 3 reveals ≤60% of answer characters).

**Client UX:**
- "Hint" button revealed inside the PuzzleCard once the reader has taken ≥1 incorrect guess OR after 30s on the puzzle (whichever first).
- Each tap reveals the next tier; button disables after hint 3.
- Hints render in a list above the input, each labeled `Hint 1:` / `Hint 2:` / `Hint 3:`.

**Why client-side reveal vs server round trip per hint:** hints are *designed* to be revealed, not protected. The answer alone is secret. A reader who opens DevTools to peek at hint 3 has done manually what tapping does anyway — and given themselves the *same* information they'd get by giving up. Net: no functional cheat surface, and we save 2 API calls per puzzle.

### Implementation surface

- `server/puzzleService.ts` — generation prompts request 3 hints; validators check hint quality.
- `shared/types/puzzles.ts` — **NEW.** `Hints = [string, string, string]`; discriminated union for `payload` by `type`.
- `client/src/components/PuzzleCard.tsx` — hint reveal state machine.

### Trade-offs

- **Pro:** Zero per-hint cost or latency.
- **Pro:** Hint quality is verifiable at generation time.
- **Con:** Hints are pre-baked, not adaptive to *how* the reader is struggling. Acceptable for v1; adaptive hints are a v2 enhancement.

## Approach 6 — Failure handling + resolution-signal threading

### Goal

A reader who can't solve a puzzle never gets stuck. Solving means *something* (small reward); skipping costs *nothing* (story continues cleanly). The next narration call must reliably know the puzzle was resolved and how.

### Design

- **Unlimited attempts** on the puzzle itself.
- **Skip Puzzle** button appears after the third hint is revealed OR after 60 seconds of inactivity on the card (whichever first). Always available once visible. Copy is "Skip Puzzle" — never "Give up."
- **On solve:** `POST /api/puzzle/attempt` with `{ puzzleId, submission }` → server validates → returns `{ correct: true }` → client shows a brief success state → next narration call's context includes `[reader solved the {type} puzzle — give a small narrative bonus appropriate to the story]`.
- **On skip:** `POST /api/puzzle/attempt` with `{ puzzleId, skip: true }` → returns `{ skipped: true }` → next narration call's context includes `[reader skipped the {type} puzzle — continue softly, do not punish, the meaning escapes them for now]`.

### Resolution-signal threading mechanism (resolves should-fix #7)

No new column on `messages` is required. The mechanism:

1. The `puzzle_attempts` table already records whether each attempt was `correct` or `skipped` (see §Data model).
2. When `aiService` builds context for the next narration call, it queries: *"for this story, list resolved puzzles (correct OR skipped) since the last narration call."* SQL: `SELECT p.type, MAX(a.correct) AS correct, MAX(a.skipped::int)::bool AS skipped FROM puzzles p JOIN puzzle_attempts a ON a.puzzle_id = p.id WHERE p.story_id = $1 AND p.id NOT IN (SELECT puzzle_id FROM puzzle_signals_consumed WHERE story_id = $1) GROUP BY p.id, p.type`.
3. For each resolved-but-unconsumed puzzle, format the resolution-signal string and append to the user-prompt context.
4. Mark consumed via insert into a tiny `puzzle_signals_consumed (puzzle_id, story_id, consumed_at)` table (FK-cascade on puzzle delete) so a single resolution signal is included exactly once.

Why not store the signal text on the puzzle row? Because the *content* of the signal is deterministic from the puzzle's `type` and the attempt outcome — no need to persist the string. Persisting only the *consumed* fact keeps the data layer clean.

### Edge: re-attempt after already resolved

`POST /api/puzzle/attempt` is idempotent in the following sense:
- If the puzzle is **already resolved** (a previous attempt has `correct=true` OR `skipped=true`), the endpoint returns the prior resolution state (`{ correct, skipped }`) and does **not** insert a new attempt row. No 409; no error. Rationale: double-tap, lost client state, or replay should be quietly safe; the puzzle has one terminal state.
- If the puzzle is **not yet resolved**, every submission inserts a new attempt row (correctness recorded; multiple incorrect attempts are interesting analytics).

### Edge: daily-spend-cap interaction mid-puzzle-generation

The daily spend cap is checked at the start of each AI call (`spendTracker.precheck()` in the existing pattern). Three cases:
- **Cap hit before narration:** narration call is blocked; no puzzle path runs.
- **Cap hit between narration and puzzle gen:** the narration response is already in flight or returned; we attempt puzzle gen. If `precheck` fails, **skip the puzzle gen entirely and fall straight through to the fallback pool** (no re-roll, no AI re-attempt). Rationale: the in-flight narration was authorized; the puzzle is a soft want, not a need.
- **Cap hit mid puzzle-gen call:** impossible at the precheck layer (precheck happens before the request); the call either runs to completion (counted against tomorrow's budget if mid-day rollover) or returns an error which falls to fallback.

### Implementation surface

- `server/routes.ts` — `POST /api/puzzle/attempt` endpoint (Zod-validated, rate-limited via new `puzzleAttemptLimiter` — see §API).
- `server/dbStorage.ts` — `recordPuzzleAttempt(puzzleId, sessionId, { submission?, skip? })`; `getUnconsumedResolutionsForStory(storyId)`; `markResolutionConsumed(puzzleId, storyId)`.
- `server/aiService.ts` — call `getUnconsumedResolutionsForStory` while building narration context; format + append to user prompt; on successful narration return, batch-call `markResolutionConsumed` for each.
- `client/src/components/PuzzleCard.tsx` — Skip button visibility timer; success / skipped end states.

### Trade-offs

- **Pro:** Zero risk of reader getting stuck. Brand stays warm.
- **Pro:** Solving still has narrative weight (the bonus), so puzzles aren't busywork.
- **Pro:** Resolution threading uses only the tables we're already building plus one tiny join table — no new columns on `messages` or `puzzles`.
- **Con:** A reader who skips every puzzle gets ~95% of the story experience. That's a feature, not a bug: puzzles are *opt-in engagement*, not *forced gameplay*. Confirmed by user in brainstorm.

## API

### `POST /api/puzzle/attempt`

**Request:**
```ts
{
  puzzleId: string,         // UUID
  submission?: string,      // present unless skip=true
  skip?: boolean,           // when true, submission must be absent
  hintsUsed: number         // tracked client-side, sent for analytics
}
```

**Response (puzzle not yet resolved, submission incorrect):**
```ts
{ correct: false, skipped: false, attemptsRemaining: null }
```

**Response (puzzle resolved this call OR already resolved by a prior call):**
```ts
{ correct: boolean, skipped: boolean }
```

The endpoint is **idempotent on the resolved state** — see §Approach 6 "Edge: re-attempt after already resolved." The response does NOT return the resolution-signal string (that flows through the narration context on the next narration call, not to the client).

**Validation (Zod):**
- Reject if both `submission` and `skip` are present.
- Require `puzzleId` to exist and to belong to a puzzle whose `session_id` matches the requesting session and whose `story_id` matches the `x-story-id` header.

**Rate limit:** existing `strictLimiter` (5/hour per session) is too tight — a single puzzle can exceed it. Add a new `puzzleAttemptLimiter` in `server/rateLimit.ts` keyed by `(sessionId, puzzleId)` allowing 30 attempts per puzzle per hour. Submissions are cheap (string compare); the limit exists only to protect against scripted abuse.

**Errors:** opaque to client per project rule (`docs/api-and-cost.md`).

## Testing

### Unit (Vitest)
- `validatePuzzle` for each type: success case, sort-mismatch fail, length-bound fail, mapping-bijection fail (cryptogram), hint-leak fail.
- Fallback-pool loader: deterministic seed test, empty-pool warning path.
- Budget calculator: `puzzleCountForStory(length, soFar) → { target, cap, canEmit }`.
- Resolver-signal threading: insert resolved puzzle, assert `getUnconsumedResolutionsForStory` returns it; call `markResolutionConsumed`, assert second call returns empty.

### Integration
- End-to-end happy path: seed story → mock narration to return `puzzle_request` → assert puzzle row created → POST attempt with correct answer → call narration again → assert resolution signal appears in injected context → assert `puzzle_signals_consumed` row exists.
- Failure path: mock generation to return invalid puzzle → assert re-roll attempted → mock second fail → assert fallback used.
- Budget gate: set `puzzle_count_so_far = puzzle_cap`, assert narration prompt-build forbids emitting `puzzle_request` (verifiable by inspecting injected context fields).
- Cross-session safety: session A creates puzzle in story A; session B POSTs `/api/puzzle/attempt` with that puzzleId; assert rejection (NOT FOUND or 403 — exact code per `docs/api-and-cost.md` convention).
- Idempotency: POST `/api/puzzle/attempt` correct → assert success → POST again with submission → assert same success response, NO new `puzzle_attempts` row.

2. Repeat: create another → encounter puzzle → skip it → continue → end → archive.
3. With `puzzles_enabled=false`: confirm no puzzles emitted, no narration regression (compare a paragraph or two with a previous puzzleless run for tone).

## Rollout

Behind an `app_config` flag, `puzzles_enabled` (default `false`).

1. **Internal validation.** Set `puzzles_enabled = true`. To force a puzzle in a single test story without waiting for organic emission, temporarily edit the `puzzle_budgets` JSON for the `25` key to `{"target": 1, "cap": 1}` and create a story whose theme practically begs for a cipher (e.g., "detective unraveling a coded note"). Validate: narration handles the resolution signal naturally; PuzzleCard renders correctly for all three types; skip path works; cost/latency match estimates. Restore the budget JSON when done.
2. **Soft launch.** Flip `puzzles_enabled = true` globally. Watch event-log for `puzzle_generated`, `puzzle_attempted`, `puzzle_skipped` event volumes. Watch error rates on `puzzleService` (validation failures, fallback uses). Watch narration latency p50/p95.
3. **Tune budgets.** Based on first-week data, adjust `puzzle_budgets` in `app_config` if puzzles feel too frequent / too rare.

**Rollback:** flip `puzzles_enabled = false`. The narration prompt's puzzle section is conditioned on the flag in the prompt-build code (not in the prompt text itself), so it becomes inert immediately. No code redeploy needed.
