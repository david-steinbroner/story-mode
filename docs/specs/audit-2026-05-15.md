### Convergence: both models attack the audit's *risk-classification confidence*

- **Gemini:** I called CSP (#19) "low risk + ~30 min." Wrong. CSP debugging in prod (Shadcn inline styles, PostHog, Sentry cross-origin, Safari quirks) is *hours*, and bundling it with the one-line queryKey fix means a CSP revert also reverts #1.
- **Qwen:** I called `staleTime: Infinity` "intentional and correct" in the verified-clean list. Partially wrong. The multi-tab desync angle is real: a reader on phone + desktop can mutate against stale `game-state` cache. (Qwen's stronger claim — that mutation failures "poison" the cache — is overstated; the existing optimistic-mutation pattern in `App.tsx:113-239` rolls back on error correctly. The multi-tab angle stands.)

Both push toward smaller, risk-isolated PRs rather than thematic buckets.

### 🔴 New finding (Qwen catch — actual audit miss)

37. **Token-volume blind spot — context bound is by message count, not token count.** `server/aiService.ts` uses `storage.getRecentMessages(sessionId, 10, storyId)` to bound AI context. Ten messages can swing from ~1.5K to ~12K input tokens depending on whether they're dialogue snippets or descriptive pages. Risks: (a) Render request timeout (default ~30-60s) on long-context calls; (b) cost scaling that's invisible in the audit because we verified only the message count; (c) the `daily_spend` pre-flight check at `routes.ts:668` IS in place, but `getRecentMessages` itself doesn't truncate by token budget. Fix shape: add lightweight `js-tiktoken` token counting to `getRecentMessages` and the summary injector; cap context at a hard token budget (e.g., 8k for Haiku) and truncate oldest messages when exceeded. Already partly compensated by the rolling-summary mechanism (every 10 unsummarized messages), but a token-aware safety net belongs in code. ~1-2h.

- **staleTime tuning** — drop from `Infinity` to `3 minutes`, set `refetchOnWindowFocus: true` and `refetchOnReconnect: true`. Addresses multi-tab desync without breaking the codebase's existing optimistic-mutation pattern. Not in PR-A but worth a follow-up PR sometime after the M6 push.
- **#15 reader_input is a behavioral change, not pure hygiene** — Gemini correctly points out that wrapping recent player messages and character bio in XML tags shifts the prompt's attention weighting. Probably harmless given Haiku's instruction-following, but warrants a qualitative smoke test (read 3-5 pages post-change to check tone hasn't drifted), not just a "does it still compile" check.

### 🟢 Pushed back on (where I think the models overshot)

- **Qwen's "staleTime poisons cache on mutation failure"** — overstated. The optimistic-mutation pattern in `App.tsx:113-239` already handles this: `onMutate` snapshots `previousMessages`, `onError` rolls back to the snapshot, leaving cache consistent with last-known server state. Not poisoned. The multi-tab desync concern is the load-bearing one.
- **Qwen's admin-toggle propagation example** — off. The admin AI model toggle (`server/aiModel.ts`) is server-side; clients don't cache it. `staleTime: Infinity` doesn't affect that propagation.
- **Gemini's "#15 should be its own PR for playtest"** — directionally right but probably overweighted for a solo project. Bundle #15 with the other AI-prompt-touching change if any; otherwise the smoke test (`tsc` + read 3 pages on localhost) handles it.
